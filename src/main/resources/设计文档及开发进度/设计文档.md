# RAG 知识助手 Demo 系统设计说明书

## 1. 项目概述与目标

本项目是一个基于 Java 语言和 Spring Boot 框架构建的单体架构 RAG 问答系统演示程序。核心目标是构建一个功能完整、模块清晰、易于扩展的知识增强型问答应用，完整覆盖从文档上传与手动录入、内容解析与向量化、混合检索与召回、上下文增强生成，到前端流式对话交互的全链路技术验证。

项目聚焦于核心流程的实现与模块化设计，不涉及用户登录、复杂权限控制、微服务治理或高并发性能优化。系统以简洁清晰的分层架构组织代码，为后续功能迭代提供可靠基础。

## 2. 技术选型说明

| 层级 | 组件 | 用途说明 |
|:-----|:-----|:---------|
| 后端框架 | Spring Boot 3.x | 应用容器、依赖注入、REST API 提供 |
| AI 抽象层 | Spring AI | 统一封装大语言模型调用、向量数据库操作、文档处理流水线及会话记忆管理 |
| 本地模型运行环境 | Ollama | 本地运行开源对话模型与嵌入模型 |
| 向量数据库 | ChromaDB | 向量存储与相似度检索，以容器化方式运行 |
| 文档解析 | Apache Tika | 从 PDF、Word、TXT 等多种文件格式中提取纯文本内容 |
| 会话存储 | H2 Database | 内嵌数据库，用于持久化对话历史记录 |
| 前端技术 | HTML + CSS + JavaScript | 原生技术栈，通过 SSE 机制接收流式响应 |

## 3. 系统整体架构

系统采用经典的分层架构设计，从上至下依次为前端界面层、控制器层、业务服务层、Spring AI 抽象层以及外部依赖层。

前端部分由两个独立页面构成，通过 HTTP 接口与后端交互，并利用 SSE 连接接收流式生成的回答内容。

控制器层负责暴露 RESTful 接口，接收前端的对话请求和知识库管理请求，并将其分发给对应的业务服务。

业务服务层包含三个主要分支：一是面向 RAG 问答的核心编排服务，负责串联检索与生成流程；二是面向知识库管理的文档处理与索引构建服务；三是面向会话管理的对话历史维护服务。

Spring AI 抽象层作为整个系统的技术基座，提供了统一的聊天客户端接口、向量存储操作接口以及会话记忆持久化接口，使得上层业务代码无需关心底层模型或数据库的具体实现差异。

外部依赖层由两个独立运行的服务组成：Ollama 提供本地模型推理能力，ChromaDB 提供向量检索能力。H2 数据库以文件形式嵌入应用进程。

## 4. 页面设计

系统前端由两个独立页面组成，均采用单页应用模式进行切换，无需整页刷新。

### 4.1 对话工作台页面

该页面是用户与系统交互的主入口，布局分为三个核心区域。

侧边栏会话区位于页面左侧，展示所有历史会话列表，每个会话项支持点击切换、删除操作，并提供一个新建会话按钮。

主消息展示区占据页面中央主要空间，以气泡形式按时间顺序展示用户问题与助手回答，回答内容支持流式逐字渲染。

底部输入区包含文本输入框与发送按钮，用户在此输入问题并发起对话请求。

顶部导航栏放置一个知识库管理入口按钮，点击后切换至知识库管理中心页面。

### 4.2 知识库管理中心页面

该页面是面向管理员的运营维护界面，用于知识条目的增删改查操作。

页面采用左右结构布局。左侧为操作面板，包含文档上传组件（支持拖拽上传与点击选择）以及手动录入表单（含标题、内容、标签等字段）。右侧为知识条目列表区，以表格形式分页展示已入库的知识片段，每条记录附有查看详情、编辑和删除操作按钮。列表上方提供关键词搜索框，支持对条目进行过滤检索。

页面顶部同样提供返回对话工作台的导航入口。

## 5. 后端模块划分

后端代码按功能边界划分为六个核心模块，各模块内部高内聚，模块间通过明确的接口或 Spring 依赖进行协作。

### 5.1 配置模块

负责集中管理系统中所有可调整的参数和常量。通过 Spring Boot 的类型安全配置绑定机制，将配置文件中以特定前缀开头的属性映射为 Java 对象。管理的配置项包括检索返回数量、相似度阈值、提示词模板、文件存储路径等。

### 5.2 会话管理模块

用于维护用户与系统之间的对话上下文。复用 Spring AI 提供的基于 JDBC 的会话记忆存储能力，利用 H2 内嵌数据库持久化每个会话的历史消息。模块对外提供会话列表查询、历史消息获取以及会话删除能力。

### 5.3 LLM 接入模块

对 Spring AI 聊天客户端的一层薄封装，负责提供统一的流式生成调用入口，并允许通过配置文件灵活切换底层模型服务。模块本身不处理业务逻辑，仅作为模型调用能力的适配出入口。

### 5.4 知识库管理模块

负责处理文档从原始文件到向量索引入库的完整流程，并支持对已入库条目的在线管理。模块细分为以下子功能单元：

- 文档解析单元：集成 Apache Tika，屏蔽不同文档格式的解析差异，对外统一输出纯文本内容。
- 索引构建单元：接收文本内容，执行分块、向量化操作，并调用向量存储接口完成索引写入。
- 条目管理单元：提供知识条目的分页查询、详情查看、内容更新及删除能力。更新操作需触发重新向量化。

### 5.5 RAG 主流程模块

系统的核心编排单元。在初始化阶段将向量存储组件以顾问形式绑定至聊天客户端。当接收到用户问题时，该顾问自动拦截请求，执行向量相似度检索，将检索到的文档片段作为上下文填充至预设提示词模板中，最后将组装完成的提示词发送至大语言模型生成答案。模块对外提供流式返回接口。

### 5.6 通用组件模块

包含统一响应结构封装、全局异常处理、工具类等公共代码，为其他模块提供基础支撑。

## 6. 后端 API 接口清单

所有接口均返回统一结构的 JSON 响应，流式接口采用 SSE 协议。

### 6.1 核心对话接口

| 名称 | 方法 | 路径 | 说明 |
|:-----|:----:|:-----|:-----|
| 流式对话 | POST | `/api/chat/stream` | 接收用户消息与会话标识，通过 SSE 流式返回 LLM 生成的答案片段 |

### 6.2 会话管理接口

| 名称 | 方法 | 路径 | 说明 |
|:-----|:----:|:-----|:-----|
| 获取会话列表 | GET | `/api/conversations` | 返回所有会话的摘要信息 |
| 获取会话历史消息 | GET | `/api/conversations/{id}/messages` | 根据会话标识返回完整消息记录 |
| 删除会话 | DELETE | `/api/conversations/{id}` | 删除指定会话及其关联消息 |

### 6.3 知识库管理接口

| 名称 | 方法 | 路径 | 说明 |
|:-----|:----:|:-----|:-----|
| 文档上传并索引 | POST | `/api/knowledge/upload` | 接收文件，执行解析、分块、向量化并入库 |
| 手动录入知识条目 | POST | `/api/knowledge/manual` | 接收标题、内容、标签，处理后入库 |
| 分页查询条目列表 | GET | `/api/knowledge/items` | 支持分页与关键词检索 |
| 获取条目详情 | GET | `/api/knowledge/items/{id}` | 返回指定条目的完整内容与元数据 |
| 更新知识条目 | PUT | `/api/knowledge/items/{id}` | 修改内容并触发重新向量化 |
| 删除知识条目 | DELETE | `/api/knowledge/items/{id}` | 从向量库中移除该条目对应数据 |

## 7. 典型业务流程

### 7.1 对话问答流程

1. 用户在对话工作台输入问题并发送，前端携带当前会话标识向流式对话接口发起连接。
2. 控制器层将请求转发至 RAG 核心编排服务。
3. 编排服务内部的聊天客户端触发绑定的检索顾问。
4. 检索顾问将问题转换为向量表示，向 ChromaDB 请求相似度最高的若干文档片段。
5. ChromaDB 返回相关文本片段列表。
6. 检索顾问将片段作为上下文变量填充至预设提示词模板，形成完整提示词。
7. 聊天客户端将提示词发送至 Ollama 模型服务。
8. Ollama 以流式方式逐令牌生成内容，经后端透传至前端 SSE 连接。
9. 前端持续接收数据流并追加渲染，直至流结束。

### 7.2 知识入库流程

文档上传入库流程：用户选择文件上传，后端接收后交由 Tika 提取纯文本，经默认分块策略切分后，调用嵌入模型生成向量，最后存入 ChromaDB 指定集合。

手动录入流程：用户在管理页面填写标题与内容并提交，后端将内容文本视为独立知识条目，执行分块、向量化后入库。

条目更新流程：用户修改条目内容后提交，后端先删除该条目原有向量数据，再对新内容重新执行分块与向量化并写入。

## 8. 配置文件关键项说明

系统主配置文件为 YAML 格式，关键配置项及其作用如下：

- Ollama 连接配置：指定本地服务地址、对话模型名称、嵌入模型名称。
- ChromaDB 连接配置：指定服务主机、端口及操作集合名称，启用自动初始化。
- 会话记忆配置：指定记忆存储类型为 JDBC，启用 H2 持久化。
- 应用自定义配置：包含检索数量、相似度阈值、提示词模板等，以统一前缀组织。

## 9. 环境启动顺序

1. 启动 ChromaDB 服务（通常使用容器运行，映射服务端口至本机）。
2. 启动 Ollama 服务，并确保所需对话模型与嵌入模型已预先拉取。
3. 使用 Maven 或 IDE 启动 Spring Boot 应用，框架会自动连接外部服务并初始化 H2 数据库表结构。
4. 打开浏览器访问应用端口，即可使用对话工作台与知识库管理中心。

## 10. 扩展性预留

尽管当前版本为功能演示版，但在架构设计上已为后续迭代预留清晰扩展点：

- 文本分块策略当前采用默认实现，未来可替换为语义分块或结构化分块组件。
- 检索流程中预留重排序接口，便于集成专用重排序模型以提升精度。
- 会话管理目前基于简单标识符区分，未来可增加用户体系与权限隔离。
- 向量数据库与 LLM 服务均通过 Spring AI 抽象层访问，更换底层实现仅需调整配置与依赖。

## 11. 开发顺序建议

后端开发按照接口为单位依次推进，优先完成核心对话闭环，再扩展知识库管理能力。前端开发先完成对话工作台页面，确保问答主流程可用，再开发知识库管理中心页面。详细的任务清单与进度跟踪由独立的后端开发进度文档和前端开发进度文档分别管理。

本文档旨在为项目开发者与协作者提供一份清晰、结构化且无歧义的全局视图，同时也可作为与大语言模型交互时快速恢复项目上下文的权威参考材料。
